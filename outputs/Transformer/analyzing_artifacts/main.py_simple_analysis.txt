# main.py – Logic Analysis

1. Import and Load Configuration 
   • Import a utility (or module) from config.py that reads config.yaml.
   • The configuration contains all hyperparameters (e.g., total_steps, warmup_steps, dropout, d_model, d_ff, etc.), dataset batching sizes (source and target token counts), and evaluation parameters (beam_size, length_penalty, max_output_length).
   • No values are invented; every parameter is taken from config.yaml.

2. Initialize the DatasetLoader
   • Create an instance of DatasetLoader with the configuration.
   • Call load_translation_data() to get the training and validation DataLoaders for the translation task.
   • Similarly (if needed), call load_parsing_data() for the constituency parsing task.
   • The DatasetLoader is responsible for:
       – Preprocessing raw text using tokenizers (using Hugging Face’s tokenizers for BPE or word-piece tokenization).
       – Grouping examples into batches that contain approximately 25,000 source tokens and 25,000 target tokens as dictated by the configuration.
   • This ensures consistent tokenization and batching across experiments.

3. Instantiate the TransformerModel
   • Using the configuration parameters (from config.model), create a TransformerModel instance.
   • The model’s sub-components are set as follows:
       – Encoder: 6 layers; each layer with multi-head self-attention (8 heads with d_k = d_v = 64) and a feed-forward network (d_model=512, d_ff=2048).
       – Decoder: 6 layers; with masked self-attention (returning only allowed positions), encoder–decoder attention and position-wise feed-forward networks.
       – Embeddings: If share_embeddings is true, then use the same weight matrix for source embedding, target embedding, and pre-softmax projection; the embeddings are scaled by √(d_model).
       – Positional encoding: Use sinusoidal functions as specified.
   • The model implementation follows the design without adding any extra public members.

4. Instantiate the Training Manager (Trainer)
   • Initialize a Trainer instance, passing the TransformerModel, the relevant data loaders (at least the translation training and validation loaders), and the configuration.
   • Trainer is in charge of the training loop which includes:
       – Forward propagation: calling model.forward(src, tgt)
       – Backpropagation and parameter updates using Adam (with β₁=0.9, β₂=0.98, ε=1e-9) and the custom learning rate schedule:
             lr = (d_model^(-0.5)) * min(step_num^(-0.5), step_num * warmup_steps^(-1.5))
       – Applying dropout (with rate 0.1 for the base model) and label smoothing (0.1) as per configuration.
       – Checkpoint management: saving checkpoints (e.g., "every 10 minutes") so that later steps can average the last few checkpoints.
   • The Trainer will also implement logging and validation steps as needed.

5. Start the Training Process
   • Call Trainer.train() to run the training loop. The training loop will run for a total of 100,000 steps as defined in config.training.total_steps.
   • The Trainer internally ensures that each training step respects the batching based on token size (source: 25K, target: 25K tokens) and that the scheduler (warmup for 4000 steps) is applied.

6. Evaluation Setup Post Training
   • Once training is complete, instantiate the Evaluation module (Evaluation class) with the trained TransformerModel and configuration.
   • For the translation task, call evaluator.evaluate_translation(test_data):
       – This method uses beam search with a beam_size set in config (4) and a length_penalty of 0.6.
       – The maximum output length is computed as (input_length + 50) per the configuration.
       – SacreBLEU is then used to compute the BLEU score.
   • Optionally, if constituency parsing evaluation is desired:
       – Call evaluator.evaluate_parsing(test_data) with the test data from parsing.
       – Note: Although the paper specifies different beam search parameters (e.g., beam size 21, length penalty 0.3, output length = input_length + 300) for parsing, the configuration file currently provides values for translation evaluation. This discrepancy should be noted and clarified (or overridden) within the Evaluation module if parsing experiments are to be performed.

7. Output and Logging of Final Metrics
   • Print or log the evaluation results (BLEU scores for translation and F1 scores for parsing).
   • These metrics will be compared with the benchmarks reported in the paper (e.g., BLEU ≈ 27.3 for the base English–German model).
   • Ensure that results are logged along with details about which checkpoints were averaged (e.g., “last 5 checkpoints” for the base model).

8. Final Flow Control and Execution
   • Wrap all the above steps into a main() function.
   • Protect the entry point with “if __name__ == '__main__': main()” so that the script can be executed directly.
   • This approach makes sure that the program is self-contained and allows for potential unit testing and modular expansion.
