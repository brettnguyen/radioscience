# Logic Analysis: `src/data/orbit_loader.py`

## 1. Purpose and Scope

`src/data/orbit_loader.py` implements the `OrbitLoader` adapter layer defined in the design. Its responsibility is to provide transmitter and receiver `StateVector` sequences at the exact high-rate timestamps of each COSMIC-2 `SignalWindow`.

This module supports the real-data reproduction pathway of the paper, where accurate GNSS transmitter and COSMIC-2 receiver geometry is required for:

- line-of-sight construction,
- tangent-point height filtering,
- candidate magnetic-field point placement along the LOS,
- BP-plane construction,
- stationary-transmitter correction,
- final geolocation from distance along the LOS.

The module does **not** perform geolocation, tangent-height filtering, BP-plane construction, or back propagation. It only loads and interpolates orbit/state data.

The public API must follow the design exactly:

```text
class OrbitLoader {
    +__init__(config: AppConfig)
    +load_receiver_orbit(leo_id: str, date: date) Any
    +load_gnss_orbit(date: date) Any
    +interpolate_rx_states(window: SignalWindow) list~StateVector~
    +interpolate_tx_states(window: SignalWindow) list~StateVector~
}
```

No additional public methods should be required by downstream modules.

---

## 2. Configuration Dependencies

This module must use configuration-derived paths and should not hard-code external data locations.

Relevant `config.yaml` entries:

```yaml
paths:
  receiver_orbit_dir: "data/cosmic2/orbits"
  gnss_orbit_clock_dir: "data/gnss"

data:
  cosmic2:
    receiver_orbits_required: true

  gnss:
    orbit_clock_source: "International GNSS Service"
    required: true
```

In the design-level `AppConfig`, these are exposed as:

```text
AppConfig.receiver_orbit_dir
AppConfig.gnss_orbit_dir
```

or equivalent fields mapped by `src/config.py`.

The loader should use:

- `config.receiver_orbit_dir` for COSMIC-2 / LEO orbit products,
- `config.gnss_orbit_dir` for IGS GNSS orbit and clock products.

The module should not assume that data exist during synthetic-only runs. Missing orbit data should raise clear errors only when real-data interpolation methods are called.

---

## 3. Scientific Requirements from the Paper

The paper requires accurate Tx/Rx geometry because the BP method depends on the GNSS–LEO line of sight during each 10-second processing interval.

Relevant paper constraints:

1. COSMIC-2 receiver positions are required for the LEO POD antenna high-rate RO observations.
2. GNSS transmitter positions are required from IGS orbit/clock products.
3. Geometry is evaluated over 10-second intervals.
4. GPS HR sampling is 50 Hz and GLONASS HR sampling is 100 Hz, but this module should not rely on those constants directly; it should interpolate to `SignalWindow.times`.
5. Output state vectors must be compatible with later modules that assume ECEF positions in meters and velocities in meters per second.

The paper’s BP geometry uses Tx and Rx positions at each sample and at the midpoint of the 10-second interval. Therefore, the interpolation must be deterministic and preserve the original observation timestamps.

---

## 4. Inputs and Outputs

### 4.1 Inputs

The primary input to interpolation methods is a `SignalWindow`.

Relevant fields from `SignalWindow`:

```text
event_id
leo_id
gnss_id
constellation
start_time
mid_time
end_time
times
rx_states
tx_states
```

The `times` array should represent the high-rate sample times for the window.

This module should expect `times` to be either:

- Python `datetime` values,
- `numpy.datetime64` values,
- or a format consistently produced by `CosmicLoader`.

If `CosmicLoader` represents times as numeric offsets, that convention must be documented there. `OrbitLoader` should convert them into absolute UTC-like datetimes before interpolation.

### 4.2 Outputs

Both interpolation methods return:

```text
list[StateVector]
```

Each `StateVector` must contain:

```text
time: datetime
position_m: Vector3
velocity_mps: Vector3
```

The returned list length must equal the number of samples in `window.times`.

The caller, usually `CosmicLoader` or `RealDataExperiments`, may assign these lists back to:

```text
window.rx_states
window.tx_states
```

The public interface returns lists and should not require a separate public attach method.

---

## 5. Internal Data Representation

Although the public return type of `load_receiver_orbit()` and `load_gnss_orbit()` is `Any` in the design, internally the module should normalize loaded orbit files into a consistent structure containing at least:

```text
times
satellite_or_spacecraft_id
positions_m
velocities_mps, optional
clock, optional
metadata
```

The internal representation may be a private dataclass, dictionary, xarray dataset, pandas DataFrame, or another local adapter object. This is intentionally hidden behind the public methods because exact CDAAC and IGS file schemas may vary.

Important convention:

- Positions must be converted to ECEF meters before producing `StateVector`.
- Velocities must be converted to ECEF meters per second.
- Time ordering must be strictly increasing after parsing.
- Duplicate timestamps should be removed or rejected deterministically.
- Missing or non-finite position samples should be rejected before interpolation.

---

## 6. File Discovery Logic

### 6.1 Receiver Orbit Files

`load_receiver_orbit(leo_id, date)` should search under:

```text
config.receiver_orbit_dir
```

for COSMIC-2 receiver orbit files matching the requested LEO spacecraft and date.

Because exact CDAAC file naming is unclear from the paper and configuration, this module should be an adapter layer. The private discovery logic should support common patterns but remain isolated from downstream modules.

Expected behavior:

1. Search deterministically using sorted paths.
2. Prefer files matching both `leo_id` and `date`.
3. If the 10-second window crosses a date boundary, interpolation may require adjacent-day coverage. The public `load_receiver_orbit(leo_id, date)` loads one date, but `interpolate_rx_states(window)` may internally call it for all dates spanned by the window.
4. If no matching file is found, raise a clear real-data error explaining:
   - requested `leo_id`,
   - requested date,
   - searched directory,
   - expected data source from configuration.

### 6.2 GNSS Orbit and Clock Files

`load_gnss_orbit(date)` should search under:

```text
config.gnss_orbit_dir
```

for IGS GNSS orbit and clock products.

The paper states GNSS orbits and clocks are from the International GNSS Service. The exact product naming is not specified in the paper or config. Therefore, the parser layer should be format-adaptive.

Expected supported products:

- SP3 precise orbit files for GNSS satellite positions.
- Optional clock files if available.

For the current design, only `StateVector` exposes position and velocity, not clock. Therefore:

- GNSS clock parsing may be stored internally for future use,
- but `interpolate_tx_states()` should return only `StateVector` objects.

Clock data should not be forced into `StateVector`, because that would change the design.

---

## 7. Parsing Responsibilities

### 7.1 COSMIC-2 Receiver Orbit Parser

The private receiver orbit parser should handle CDAAC receiver orbit products. Exact file schema is unclear, so the parser should be implemented defensively.

It should extract:

- spacecraft identifier,
- timestamp,
- ECEF position,
- velocity if available,
- coordinate frame metadata if present.

If velocity is not available, it must be derived from interpolated position time series or finite differences in a documented deterministic way.

Required normalization:

```text
position -> meters
velocity -> meters per second
time -> datetime-compatible UTC representation
```

The parser should validate that the receiver orbit corresponds to the requested `leo_id`.

### 7.2 GNSS Orbit Parser

The private GNSS orbit parser should handle IGS orbit products, especially SP3-like products.

It should extract:

- GNSS satellite ID / PRN,
- timestamp,
- satellite position,
- optional satellite clock correction.

SP3 positions are commonly stored in kilometers, so unit conversion to meters is required if using SP3. The module should not expose SP3-native units to downstream geometry code.

The parser must support the `window.gnss_id` convention used by `CosmicLoader`, for example GPS or GLONASS identifiers. Exact mapping between file satellite identifiers and `SignalWindow.gnss_id` should be isolated in this module.

---

## 8. Time Handling

Accurate time handling is critical because the BP algorithm uses Tx/Rx positions at each high-rate sample.

The module should:

1. Convert orbit file timestamps to a consistent datetime representation.
2. Convert `SignalWindow.times` to the same representation.
3. Preserve sample order.
4. Return `StateVector.time` values matching the high-rate sample times.
5. Avoid silent extrapolation outside orbit coverage.

The paper does not specify time-scale details such as GPS time versus UTC handling. Because IGS products and COSMIC-2 files may use different time scales, this module should use `astropy.time` or equivalent reliable time conversion utilities where needed.

If time-scale metadata are absent, the implementation should document the assumption made by the parser.

---

## 9. Interpolation Logic

### 9.1 General Interpolation Requirements

`interpolate_rx_states(window)` and `interpolate_tx_states(window)` should interpolate orbit positions and velocities to every timestamp in `window.times`.

The interpolation should satisfy:

- output length equals input sample count,
- output order matches `window.times`,
- positions are ECEF meters,
- velocities are ECEF meters per second,
- no hidden downsampling,
- no hidden smoothing of the high-rate signal.

The paper does not define a specific orbit interpolation method. Therefore, this is an implementation detail, not a paper hyperparameter. The method should be deterministic and documented.

A practical approach is:

- use scipy interpolation over the loaded orbit samples,
- interpolate each ECEF component independently,
- compute velocities either from orbit-provided velocity data or from the derivative of the position interpolation.

No scientific thresholds should be hard-coded here. If extrapolation would be needed, the method should fail clearly rather than silently extending the orbit.

### 9.2 Receiver State Interpolation

`interpolate_rx_states(window)` logic:

1. If `window.rx_states` already exists and has the same length as `window.times`, validate and return it.
2. Determine dates required by `window.start_time`, `window.end_time`, and `window.times`.
3. Load receiver orbit data for `window.leo_id` and required date coverage using `load_receiver_orbit()`.
4. Merge adjacent loaded orbit segments if needed.
5. Validate coverage over all high-rate timestamps.
6. Interpolate receiver ECEF position.
7. Interpolate or derive receiver ECEF velocity.
8. Build one `StateVector` per high-rate sample.

This method should not compute tangent height. That belongs to `LosGeometry`.

### 9.3 Transmitter State Interpolation

`interpolate_tx_states(window)` logic:

1. If `window.tx_states` already exists and has the same length as `window.times`, validate and return it.
2. Determine dates required by `window.start_time`, `window.end_time`, and `window.times`.
3. Load GNSS orbit data for required dates using `load_gnss_orbit()`.
4. Select the satellite corresponding to `window.gnss_id`.
5. Validate that the selected satellite has coverage over the full window.
6. Interpolate transmitter ECEF position.
7. Interpolate or derive transmitter ECEF velocity.
8. Build one `StateVector` per high-rate sample.

If `window.gnss_id` is not present in the GNSS orbit product, the method should raise a clear error identifying the missing satellite.

---

## 10. Coordinate Frame Requirements

Downstream modules assume ECEF coordinates.

Therefore, `OrbitLoader` must ensure:

```text
StateVector.position_m -> ECEF meters
StateVector.velocity_mps -> ECEF meters per second
```

If a loaded product is already in an Earth-fixed terrestrial frame compatible with ECEF, it may be used directly after unit conversion.

If a product is in another frame, the parser must transform it before returning states. The paper does not discuss frame transformations, so any such transformation must be documented in the parser implementation.

The module should not return geodetic latitude, longitude, or altitude. Those are handled by `CoordinateTransformer`.

---

## 11. Interaction with `CosmicLoader`

The task allows either of two workflows:

1. `CosmicLoader` loads event data and calls `OrbitLoader` to populate `SignalWindow.rx_states` and `SignalWindow.tx_states`.
2. `RealDataExperiments` loads windows first and later calls interpolation before geolocation.

The design sequence diagram shows `CosmicLoader` cooperating with `OrbitLoader`.

To support both without adding public methods:

- `interpolate_rx_states(window)` returns the receiver state list.
- `interpolate_tx_states(window)` returns the transmitter state list.
- The caller assigns those lists to the `SignalWindow`.

This preserves the design interface.

---

## 12. Error Handling

The module should fail early and clearly for real-data processing errors.

Important error cases:

1. Receiver orbit directory missing.
2. GNSS orbit directory missing.
3. No receiver orbit file found for `leo_id` and date.
4. No GNSS orbit file found for date.
5. Requested GNSS satellite absent from orbit product.
6. Orbit coverage does not include all high-rate timestamps.
7. File parser cannot identify units or coordinate frame.
8. Parsed positions contain NaN or non-finite values.
9. Interpolated states have invalid shape.
10. Returned state count does not match `len(window.times)`.

Synthetic mode should not require orbit data, but this module is only invoked during real-data processing.

---

## 13. Caching Strategy

Orbit files are reused across many 10-second windows. To keep full-year processing efficient, this module should cache loaded orbit products internally.

Recommended internal caches:

```text
receiver orbit cache keyed by (leo_id, date)
gnss orbit cache keyed by date
```

This cache should be private to `OrbitLoader`.

Caching must be deterministic and should not alter results.

No cache size is specified in `config.yaml`, so the initial implementation can use simple in-memory caching and document memory implications.

---

## 14. Reproducibility Considerations

The paper processes large COSMIC-2 datasets over 2021 and 2023. Reproducible orbit handling requires:

1. Deterministic file discovery order.
2. Deterministic interpolation.
3. Explicit unit conversion.
4. Explicit time-scale handling.
5. Clear logging or error messages for missing orbit data.
6. No silent fallback to approximate broadcast orbits unless explicitly configured in the future.

The configuration specifies IGS orbit/clock products as the GNSS source, so this module should not silently substitute another source.

---

## 15. Validation Checks

Before using this module in full BP processing, the following checks should be performed.

### 15.1 Shape and Unit Checks

For a sample `SignalWindow`:

- `len(rx_states) == len(window.times)`
- `len(tx_states) == len(window.times)`
- each position norm is physically plausible:
  - COSMIC-2 receiver radius should be Earth radius plus roughly LEO altitude,
  - GNSS transmitter radius should be GNSS orbital scale.
- velocities are finite and in meters per second.

These checks should not impose paper-level QC thresholds; they are implementation sanity checks.

### 15.2 LOS Consistency Check

After interpolation, downstream `LosGeometry` should verify that candidate points along the LOS lie between Rx and Tx using the sign convention adopted there.

This is important because `config.yaml` notes:

```yaml
back_propagation:
  geolocation:
    note: >
      The sign convention must be verified during implementation so that the
      geolocation lies along the receiver-transmitter line of sight.
```

`OrbitLoader` supplies the positions needed for that validation.

### 15.3 Time Boundary Check

Test windows near midnight UTC to ensure adjacent-day orbit files are loaded when necessary.

### 15.4 Satellite Identifier Check

Test both GPS and GLONASS identifiers because the configuration states:

```yaml
data:
  cosmic2:
    sampling_rates_hz:
      gps: 50
      glonass: 100
```

The interpolation should be independent of sampling rate and rely only on timestamps.

---

## 16. Relationship to Paper Methodology

This module directly supports Sections 3 and 4 of the paper:

- Section 3 requires real Tx/Rx observational geometry for BP.
- Section 4 applies BP to COSMIC-2 HR phase and amplitude data in 10-second intervals.

The stationary-transmitter correction later uses sample-wise Tx/Rx projected coordinates:

```text
(x1, z1) for Tx
(x2, z2) for Rx
```

Those are computed later from the `StateVector` lists produced here.

Therefore, any interpolation error in this module propagates into:

- LOS direction,
- tangent-point height,
- magnetic-field candidate point,
- scan angle,
- phase correction path lengths,
- final geolocation distance and position.

For that reason, this module should prioritize correctness, deterministic behavior, and explicit failure over permissive fallback behavior.

---

## 17. What This Module Should Not Do

`src/data/orbit_loader.py` should not:

1. Compute scintillation indices.
2. Detrend phase.
3. Normalize SNR or amplitude.
4. Compute tangent-point height.
5. Build BP planes.
6. Evaluate IGRF magnetic field.
7. Perform stationary-transmitter correction.
8. Run back propagation.
9. Apply geolocation QC.
10. Write output files.
11. Plot diagnostics.

Those functions belong to other modules in the design.

---

## 18. Ambiguities to Document During Implementation

The paper and configuration leave several orbit-related details unresolved:

1. Exact COSMIC-2 receiver orbit file schema.
2. Exact CDAAC orbit product naming convention.
3. Whether receiver orbit files include velocity directly.
4. Exact IGS GNSS orbit and clock product type used operationally.
5. Exact time-scale conventions in CDAAC HR files and IGS products.
6. Whether clock corrections are needed for later phase reconstruction in the chosen data product.
7. Exact satellite identifier mapping between COSMIC-2 metadata and IGS orbit files.

These ambiguities should be isolated inside `OrbitLoader` private parsers so the rest of the pipeline continues to use the stable `StateVector` interface.

---

## 19. Expected Final Behavior

When implemented correctly, `OrbitLoader` allows the real-data pipeline to do the following:

1. `CosmicLoader` reads a COSMIC-2 high-rate event.
2. The event is segmented into 10-second `SignalWindow` objects.
3. `OrbitLoader.interpolate_rx_states(window)` returns receiver states at every high-rate sample.
4. `OrbitLoader.interpolate_tx_states(window)` returns GNSS transmitter states at every high-rate sample.
5. The populated `SignalWindow` is passed to `Geolocator`.
6. Later modules reproduce the paper’s BP geolocation workflow using accurate sample-wise Tx/Rx geometry.

The module is therefore a foundational adapter between external orbit data products and the paper’s real-geometry back-propagation algorithm.
