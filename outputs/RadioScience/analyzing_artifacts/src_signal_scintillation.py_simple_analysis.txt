## Logic Analysis: `src/signal/scintillation.py`

### 1. Purpose of this Module

`src/signal/scintillation.py` implements the `ScintillationCalculator` specified in the design. Its role is narrowly defined:

- Compute the 10-second phase scintillation index:

\[
\sigma_\phi = \mathrm{std}(\phi)
\]

- Compute the amplitude scintillation index:

\[
S_4 =
\frac{
\sqrt{\langle I^2\rangle - \langle I\rangle^2}
}{
\langle I\rangle
}
\]

where:

\[
I = A^2
\]

- Compute mean SNR for diagnostics.
- Return a `ScintillationMetrics` object.
- Set the `passes_threshold` flag using the paper/configuration threshold:

\[
\sigma_\phi > 0.25 \ \mathrm{rad}
\]

This module is used by `Geolocator.process_window()` before back propagation. It supports the pre-BP quality-control stage described in the paper’s Section 4.

---

### 2. Paper Alignment

The paper states that COSMIC-2 high-rate scintillation data are processed in 10-second intervals. For each interval:

- Phase scintillation index \(\sigma_\phi\) is computed in ground processing.
- Back propagation is attempted only when:

\[
\sigma_\phi > 0.25 \ \mathrm{rad}
\]

- Amplitude scintillation is represented by \(S_4\).
- The amplitude input is COSMIC-2 SNR scaled to a 1-Hz band.
- The paper reports sanity-check averages over accepted geolocations:

\[
\langle \sigma_\phi \rangle \approx 1.65 \ \mathrm{rad}
\]

\[
\langle S_4 \rangle \approx 0.32
\]

\[
\langle \sigma_\phi / S_4 \rangle \approx 5.25 \ \mathrm{rad}
\]

This module should reproduce the definitions used to generate those quantities, while leaving higher-level QC to `QualityController`.

```yaml

```

The implementation should document that `QualityController` is the authoritative configurable QC layer.

### 5. Data Assumptions

#### 5.1 Phase Input

`SignalWindow.phase_rad` should already be prepared by `SignalPreprocessor.prepare_window()`.

According to the program call flow:

```text
Geolocator.process_window()
    -> SignalPreprocessor.prepare_window()
    -> ScintillationCalculator.compute()
```

Therefore, in this module:

- Do **not** detrend phase again.
- Do **not** unwrap phase again unless the implementation chooses to defensively handle obvious wrapping, but this should normally belong to `SignalPreprocessor`.
- Treat `phase_rad` as the detrended phase fluctuation series used for \(\sigma_\phi\).

This matters because the paper’s exact detrending method is unspecified, and `config.yaml` explicitly records this ambiguity:

```yaml
unclear_from_paper:
  - "Exact phase detrending method before sigma_phi calculation."
```

So this module should not silently impose a second detrending method.

#### 5.2 Amplitude Input

`SignalWindow.amplitude` should represent the amplitude used for scintillation calculations.

The paper states:

- COSMIC-2 amplitude is represented by SNR scaled to a 1-Hz band.
- \(S_4\) is based on intensity fluctuations.
- Intensity should be computed as:

\[
I = A^2
\]

Therefore:

- Use `window.amplitude` for \(A\).
- Compute \(I = A^2\).
- Do not convert to dB.
- Do not square SNR again if amplitude has already been converted to intensity elsewhere.
- The expected upstream convention is that `SignalPreprocessor.normalize_amplitude()` preserves the amplitude meaning required by this formula.

#### 5.3 SNR Input

`SignalWindow.snr_vv` should be used to compute:

\[
\overline{\mathrm{SNR}}
\]

If `snr_vv` is unavailable or empty, fallback logic may use `amplitude` only if `amplitude` is explicitly the SNR-like amplitude from the COSMIC-2 product. This fallback should be documented because the paper states amplitude is represented by SNR.

### 6. Method Logic

---

## 6.1 `compute_sigma_phi(phase_rad)`

### Input

```text
phase_rad: ndarray
```

Expected units:

```text
radians
```

### Formula

The paper defines the phase scintillation index as the standard deviation of detrended phase:

\[
\sigma_\phi = \sqrt{\langle \phi^2\rangle - \langle \phi\rangle^2}
\]

Implementation should use the population standard deviation, equivalent to:

```text
np.nanstd(phase_rad, ddof=0)
```

Rationale:

- The paper uses angle brackets \(\langle \cdot \rangle\), implying interval averages rather than sample-estimator correction.
- No `ddof=1` correction is specified.
- The 10-second windows contain many high-rate samples, so the difference is small, but `ddof=0` is more faithful to the paper’s notation.

### NaN Handling

Use only finite samples.

Recommended logic:

1. Convert input to NumPy array.
2. Select finite values using `np.isfinite`.
3. If no finite values exist, return `np.nan`.
4. Otherwise compute standard deviation with `ddof=0`.

### Edge Cases

| Case | Expected Behavior |
|---|---|
| Empty phase array | return `np.nan` |
| All NaN/Inf | return `np.nan` |
| Single finite sample | return `0.0` |
| Constant phase | return `0.0` |
| Valid detrended scintillation phase | return finite positive float |

---

## 6.2 `compute_s4(amplitude)`

### Input

```text
amplitude: ndarray
```

Expected meaning:

```text
signal amplitude A, represented by COSMIC-2 SNR scaled to 1-Hz band
```

### Formula

From `config.yaml` and the paper:

\[
I = A^2
\]

\[
S_4 =
\frac{
\sqrt{\langle I^2\rangle - \langle I\rangle^2}
}{
\langle I\rangle
}
\]

Equivalent computational steps:

1. Filter finite amplitude samples.
2. Reject or ignore invalid non-finite values.
3. Compute:

\[
I_i = A_i^2
\]

4. Compute:

\[
\mu_I = \langle I\rangle
\]

5. Compute:

\[
\mu_{I^2} = \langle I^2\rangle
\]

6. Compute variance-like term:

\[
\sigma_I^2 = \mu_{I^2} - \mu_I^2
\]

7. Clip small negative roundoff values to zero:

\[
\sigma_I^2 = \max(\sigma_I^2, 0)
\]

8. Return:

\[
S_4 = \frac{\sqrt{\sigma_I^2}}{\mu_I}
\]

### Why Clip Negative Values?

Due to floating-point precision, nearly constant intensity may produce:

\[
\mu_{I^2} - \mu_I^2 < 0
\]

by a very small amount. Since physical variance cannot be negative, clip to zero.

### Mean Intensity Check

If:

\[
\mu_I \leq 0
\]

then \(S_4\) is undefined. Return `np.nan`.

Amplitude/SNR should normally be positive, but corrupted or pre-normalized data may produce zeros.

### NaN Handling

Use only finite amplitude samples.

Potentially also require:

\[
A > 0
\]

because amplitude/SNR should be positive. If non-positive finite values occur, the implementation should either:

- exclude them, or
- include them after squaring.

For physical consistency with SNR amplitude, excluding non-positive values is safer. However, if amplitude has been normalized and mean-centered upstream, negative values would indicate incorrect preprocessing for this module. In that case, returning `np.nan` or logging a warning would be appropriate.

The preferred logic is:

- Require finite values.
- Prefer positive amplitude samples.
- If no positive finite values exist, return `np.nan`.

### Edge Cases

| Case | Expected Behavior |
|---|---|
| Empty amplitude array | return `np.nan` |
| All NaN/Inf | return `np.nan` |
| All zero amplitude | return `np.nan` |
| Constant positive amplitude | return `0.0` |
| Valid scintillating amplitude | return finite nonnegative float |

## 6.3 `compute(window)`

### Input

```text
window: SignalWindow
```

Expected state:

- 10-second segment.
- Phase already detrended by `SignalPreprocessor`.
- Amplitude normalized/prepared by `SignalPreprocessor`.
- `snr_vv` still available for mean SNR diagnostics.

### Steps

1. Read:

```text
phase = window.phase_rad
amplitude = window.amplitude
snr = window.snr_vv
```

2. Compute:

\[
\sigma_\phi = \mathrm{compute\_sigma\_phi}(phase)
\]

3. Compute:

\[
S_4 = \mathrm{compute\_s4}(amplitude)
\]

4. Compute mean SNR:

\[
\overline{\mathrm{SNR}} = \langle \mathrm{snr\_vv} \rangle
\]

using finite values only.

5. If `window.snr_vv` is unavailable, empty, or all non-finite:

- Use `window.amplitude` as fallback only if it represents SNR-like amplitude.
- Otherwise return `np.nan` for mean SNR.

6. Set:

\[
\text{passes\_threshold} =
\sigma_\phi > 0.25
\]

using the paper/config threshold:

```yaml
processing.scintillation_indices.sigma_phi_threshold_rad: 0.25
```

7. Return:

```text
ScintillationMetrics(
    sigma_phi_rad=sigma_phi,
    s4=s4,
    mean_snr_vv=mean_snr,
    passes_threshold=passes_threshold
)
```

### Important QC Separation

Although `passes_threshold` is populated here, the full pre-BP QC is performed by:

```text
QualityController.passes_pre_bp(metrics, tangent_height_km)
```

That method must apply both:

\[
\sigma_\phi > 0.25 \ \mathrm{rad}
\]

and:

\[
h_{\tan} > 150 \ \mathrm{km}
\]

This module only computes scintillation metrics.

---

### 7. Relationship to the Paper’s Processing Pipeline

The paper’s processing sequence is:

1. Segment high-rate data into 10-second intervals.
2. Compute \(\sigma_\phi\) and \(S_4\).
3. Continue to BP only if:

\[
\sigma_\phi > 0.25 \ \mathrm{rad}
\]

4. Apply tangent-height filter:

\[
h_{\tan} > 150 \ \mathrm{km}
\]

5. Run BP and final QC.

This module implements step 2 and the scintillation part of step 3.

It does not implement:

- Tangent point filtering.
- Back propagation.
- \(Q\) computation.
- \(\cos\alpha\) filtering.
- Multi-valued geolocation rejection.

Those belong to `QualityController`, `BackpropProcessor`, and `Geolocator`.

---

### 8. Expected Numerical Behavior

For correctly processed COSMIC-2 accepted geolocations, aggregate diagnostics should be close to the values reported in the paper:

| Quantity | Paper Value |
|---|---:|
| Mean accepted \(\sigma_\phi\) | approximately `1.65 rad` |
| Mean accepted \(S_4\) | approximately `0.32` |
| Mean accepted \(\sigma_\phi/S_4\) | approximately `5.25 rad` |

These are not per-window acceptance criteria. They are sanity checks over a large accepted population.

The implementation should make it easy for downstream diagnostics to compute:

\[
\frac{\sigma_\phi}{S_4}
\]

but this module does not need to add another field to `ScintillationMetrics`, because the design does not include it.

---

### 9. Handling Ambiguities from the Paper

The paper does not fully specify the detrending method used before \(\sigma_\phi\). `config.yaml` records:

```yaml
phase_detrending:
  method: "unspecified_in_paper"
  default_implementation: "polynomial"
  polynomial_order: 1
```

Therefore:

- `ScintillationCalculator` should not perform detrending.
- It should assume `SignalPreprocessor.detrend_phase()` already produced the phase used for scintillation calculations.
- Sensitivity to detrending order belongs to `src/signal/preprocessing.py` and experiment-level diagnostics.

The paper also does not fully specify the exact amplitude normalization before BP. For this module, the relevant definition is explicit enough:

\[
I = A^2
\]

where \(A\) is the amplitude/SNR-like signal. Therefore `compute_s4()` should use the array supplied in `SignalWindow.amplitude`.

### 12. Implementation Risks

#### Risk 1: Double Detrending

If this module detrends phase again, \(\sigma_\phi\) may be underestimated. Avoid this.

#### Risk 2: Misinterpreting Amplitude as Intensity

If `window.amplitude` is already intensity and `compute_s4()` squares it again, \(S_4\) will be wrong. The project convention must remain:

```text
SignalWindow.amplitude = A
I = A^2
```

#### Risk 3: Using `ddof=1`

The paper uses ensemble/window averages. Use `ddof=0`.

#### Risk 4: Threshold Configuration Conflict

The design does not inject `AppConfig` into `ScintillationCalculator`, but `ScintillationMetrics` contains `passes_threshold`.

Resolution:

- Use the explicitly configured/paper threshold `0.25 rad` for the flag.
- Keep authoritative configurable QC in `QualityController`.
- Document this in the module.

#### Risk 5: Non-finite Data

COSMIC-2 files may contain missing samples, fill values, or corrupted values. All calculations should use finite values only.
